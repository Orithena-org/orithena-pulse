# Lessons from OpenAI Harness: Zero-Manual-Code Production Engineering

Source: OpenAI Engineering Blog - Harness team case study
Date captured: 2026-03-05

## What Happened

OpenAI's Harness team shipped a 1M-line production product in 5 months using zero manually-written code. Every line was generated by Codex agents. Humans designed the system, wrote specs, built feedback loops, and reviewed output -- but never wrote code directly.

This is the clearest evidence yet that the ~10x velocity claim for agent-assisted engineering is real and achievable in production (not just demos).

## Key Lessons

### 1. Humans design environments, agents write code

The Harness team's workflow: humans write intent documents, specs, and environment setup. Agents translate those into code. This inverts the traditional model where humans write code and machines run it.

**Orithena mapping:** This is already how Scout operates -- `brain/system_prompt.md`, `memory/strategy.md`, and task queue entries are the "intent documents." The Discord bot + Claude Code subprocess is the execution loop. The lesson is to invest more in spec quality and less in manual code changes. Every human session should focus on refining specs, not writing patches.

### 2. Repo legibility is a first-class concern

Repos must be navigable by agents, not just humans. This means:
- Clear, descriptive file and directory names (no clever abbreviations)
- Domain context embedded in files (docstrings that explain *why*, not just *what*)
- Architecture that's discoverable -- an agent should be able to understand the system by reading a few key files
- CLAUDE.md / README files that serve as an agent's entry point to the codebase

**Orithena mapping:** We already do this well with `CLAUDE.md` files in every repo, `NORTH_STAR.md` for organizational context, and `features/index.md` as a living architecture doc. The gap: our domain YAML files (`domains/agentic.yaml`) and scraper adapters could benefit from more inline context about *why* certain sources/thresholds were chosen. Agent-readable architecture means every config file should justify its own existence.

### 3. Observability-first: agents need feedback loops to self-correct

The biggest bottleneck in agent-driven development isn't code generation -- it's the agent's ability to see whether its code works. Agents need access to:
- Logs (did it crash?)
- Metrics (is it performing?)
- UI feedback (does it look right?)
- Test results (does it pass?)

Without these, agents generate code in the dark and require human intervention to close the loop.

**Orithena mapping:** Scout currently gets limited feedback. The Discord bot captures run success/failure and posts summaries, but Scout doesn't see:
- Whether the madison-events site actually renders correctly after changes
- Whether Pulse scraping runs produce good results
- CI/CD pipeline outcomes in real-time
- Actual user engagement with shipped products

Closing these feedback loops is the highest-leverage investment for Orithena's velocity. The `make preview` system is a start, but agents need programmatic access to preview results, not just humans viewing them in a browser.

### 4. ~10x velocity is real but conditional

The 10x comes from parallelism (multiple agents working simultaneously) and elimination of context-switching overhead (agents don't get distracted, forget context, or need coffee breaks). But it requires:
- High-quality specs (garbage in, garbage out -- faster)
- Strong test coverage (agents can self-verify)
- Clear boundaries (agents need well-defined scope, not vague "make it better")

**Orithena mapping:** Scout's task queue (`orithena-org/tasks/queue.json`) already enforces clear boundaries per run. The Actor-Critic loop (Scout builds, Critic reviews) provides a verification layer. To unlock more velocity: invest in automated test suites for both products, and ensure every task in the queue has acceptance criteria, not just a description.

### 5. Memory and context management are the real differentiators

The Harness team's agents maintained project context across sessions. This is what separates toy demos from production agent systems. Persistent memory lets agents:
- Build on previous work without re-discovering the codebase
- Learn from past failures (don't repeat the same bug)
- Maintain architectural consistency across changes

**Orithena mapping:** This is Orithena's core thesis -- "the organizational memory IS the organization" (NORTH_STAR.md). Our memory system (`context.json`, `learnings.md`, `strategy.md`, `decisions/`, `lessons.md`, `features/index.md`) is already more sophisticated than what most agent systems have. The lesson reinforces that we should continue investing in memory quality over code quantity. Every run should leave the memory better than it found it.

## Action Items for Orithena

1. **Spec-first workflow:** Ensure every task in queue.json includes acceptance criteria and context, not just a title. Humans should spend time on specs, not code.

2. **Agent-readable configs:** Add inline comments to domain YAML files explaining *why* thresholds and sources were chosen. Make every config file self-documenting for agents.

3. **Close the feedback loop:** Give Scout programmatic access to:
   - CI/CD results (GitHub Actions status)
   - Preview server output (screenshot or HTML snapshot)
   - Scraper run metrics (items found, scores, errors)
   - Site analytics (once deployed)

4. **Test coverage:** Add integration tests for both madison-events and orithena-pulse that agents can run to verify their changes.

5. **Memory hygiene:** Continue the current pattern of updating memory every run. Consider adding a "lessons from this run" field to run_output.json that gets automatically appended to org lessons.

## Bottom Line

The Harness case validates Orithena's architecture. We're already doing the right things (persistent memory, spec-driven tasks, actor-critic loop, agent-readable repos). The biggest gap is observability -- closing the feedback loop so Scout can see the impact of its changes without human intervention. That's the multiplier between where we are and 10x velocity.
